# Chromebook・Fireタブレット対応の画像仕様とPWA最適化戦略

両デバイス（Chromebook、Fireタブレット）向けに、学習教材画像を最適化するための技術仕様と実装戦略をまとめます。

## WebP形式の推奨仕様（解像度とファイルサイズ）

### 圧縮設定

文字や図解、アイコンが含まれる教材イラストには、輪郭やテキストを鮮明に保つため「**WebP（ロスレス圧縮）**」が推奨されます。

ロスレスWebPは同等のPNGファイルより約26%サイズを削減できます。

写真などの場合は「**ロッシー（非可逆）圧縮**」を品質75～85%に設定することで、画質を保ちつつファイルサイズを大幅に削減できます。

### 解像度

両端末の最大解像度（Chromebook: 1280x800、Fireタブレット: 1024x768）に合わせ、等倍表示される最大サイズ（全画面なら1280幅）を超えないように画像をリサイズしてから配信することがメモリ節約の基本となります。

---

## 小学生向けの最適なフォントサイズ（pt単位）

文部科学省の教科書基準などに照らし合わせると、紙面やデジタル画面での視認性確保のために以下のサイズが目安とされています。

### 小学低学年（1～3年生）
**26pt程度が基本**です。

### 小学中学年～高学年（4～6年生）
**22pt程度が推奨**されます。

---

## コントラスト比（WCAG 2.2基準）

### 最低基準（Level AA）

通常サイズのテキスト：**「4.5:1」**

サイズの大きなテキスト（24px以上、または18.66px以上の太字）：**「3:1」**

のコントラスト比が必須となります。

### より高い基準（Level AAA）

より多様な視覚特性に対応するためには、通常テキストで「7:1」、大きなテキストで「4.5:1」を目指すことが推奨されます。

---

## 低スペックタブレット（2GB RAM程度）での読み込み最適化

Fireタブレット（RAM 1GB～2GBモデル）などでは、メモリ不足によりブラウザ（Silk）が再読み込みや強制終了を引き起こす事象が頻発します。これを防ぐ対策は以下の通りです。

### 遅延読み込み（Lazy Loading）

画面外にある画像の遅延読み込み（Lazy Loading）を徹底し、初期ロード時のメインスレッドの負荷とメモリ消費を抑えます。

### ビットマップメモリの削減

アプリやWebビュー側で画像を扱う際、透過処理が不要な画像であればピクセルあたり4バイト消費する「ARGB_8888」ではなく、2バイトで済む「RGB_565」フォーマットを使用することで、画像のメモリ消費を半減できます。

### キャッシュの解放

WebViewを利用するアプリの場合は、適宜 clearCache 等を呼び出して不要になったリソースを解放し、メモリ不足に備えます。

## オフライン時の画像キャッシング戦略（PWA）

### Cache Storage APIの活用

ネットワークが不安定な環境でも動作させるため、Service Workerを利用してHTML、CSS、JS、および静的な画像アセットを「Cache Storage API」でデバイスに直接キャッシュします。

### 動的データの保存

オフライン時の学習進捗データや、ユーザーがアップロードする予定のファイル（手書き解答の画像など）は、非同期で動作する「IndexedDB」に保存し、オンライン復帰時にサーバーへ送信する構成が適しています。

### ストレージ制限の考慮

Chromeベースのブラウザでは、利用可能なディスク空き容量の最大33%をオフラインキャッシュとして利用でき、各オリジン（サイト）にはその約20%が上限として割り当てられます。

不要になった古い学習単元の画像キャッシュは、Service Workerの更新時などに適切に削除して容量を確保する設計が必要です。

## Day 1-5 実装への直結した仕様

### Day 4 テンプレート画像生成時の実装仕様

| 項目 | 仕様 |
|------|------|
| **ファイル形式** | WebP（図解・アイコン：ロスレス、写真：ロッシー品質75-85%） |
| **解像度** | 1280px（Chromebook基準）、アスペクト比 16:10 |
| **フォントサイズ** | 小1-3: 26pt、小4-6: 22pt |
| **コントラスト比** | Level AAA推奨（7:1 通常テキスト） |
| **初期ロード** | Lazy Loading で画面外画像を遅延読み込み |
| **キャッシング** | Service Worker で静的アセット永続化 |

### 低スペック端末対応

- Lazy Loading 必須（Firebase/Fireタブレットでのメモリ不足対策）
- RGB_565 検討（画像メモリを半減）
- Service Worker clearCache で定期リソース解放

### オフライン対応（週末運用向け）

- Cache Storage API で教材画像をデバイスに永続化
- IndexedDB で学習進捗をオフラインで記録
- オンライン復帰時に Gemini API に同期
